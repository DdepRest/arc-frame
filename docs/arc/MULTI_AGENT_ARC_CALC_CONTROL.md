# Multi-Agent Control Plan — A.R.C. / расчёты товаров, размеров, Anwis, КП, завод

Этот документ адаптирует AI-самоконтроль под проект, который делают разные агенты, а не один Gemini.

Проект уже имеет существующую структуру:

```text
AGENT.md
docs/arc/
  MULTI_AGENT_ARC_CALC_CONTROL.md
  CURRENT_STATE.md
  CALCULATION_LOGIC.md
  CALCULATION_TEST_CASES.md
  GOTCHAS.md
  RELEASE_PROCESS.md
  AUTO_UPDATE.md
```

Главная задача: сделать так, чтобы любой агент, входящий в проект, не ломал расчёты, размеры, Anwis-логику, КП, заводские размеры, релизный процесс и автообновление.

---

## 1. Источник истины для всех агентов

Единственный канонический master-файл для всех агентов:

```text
docs/arc/MULTI_AGENT_ARC_CALC_CONTROL.md
```

Этот файл является source of truth для поведения AI-агентов в проекте.

Файлы в репозитории:

```text
AGENT.md
AGENTS.md
CLAUDE.md
GEMINI.md
.cursor/rules/*
.github/copilot-instructions.md
~/.claude/skills/MULTI_AGENT_ARC_CALC_CONTROL.md
```

являются только совместимыми thin wrappers / redirect-файлами.

Они не должны дублировать правила проекта.

Если wrapper-файл противоречит `MULTI_AGENT_ARC_CALC_CONTROL.md`, приоритет всегда у:

```text
docs/arc/MULTI_AGENT_ARC_CALC_CONTROL.md
```

Проектная память и фактическое состояние проекта находятся в:

```text
docs/arc/
```

Главный файл текущего состояния проекта:

```text
docs/arc/CURRENT_STATE.md
```

## 2. Обязательный вход любого агента

Перед любой нетривиальной задачей агент обязан прочитать:

```text
docs/arc/MULTI_AGENT_ARC_CALC_CONTROL.md
docs/arc/CURRENT_STATE.md
```

Если задача касается расчётов, цен, размеров, Anwis, КП, завода, монтажа или итогов, дополнительно обязательно прочитать:

```text
docs/arc/CALCULATION_LOGIC.md
docs/arc/CALCULATION_TEST_CASES.md
docs/arc/GOTCHAS.md
```

Если задача касается релиза, версии, публикации или автообновления:

```text
docs/arc/RELEASE_PROCESS.md
docs/arc/AUTO_UPDATE.md
```

Если агент физически не имеет доступа к:

```text
docs/arc/MULTI_AGENT_ARC_CALC_CONTROL.md
```

он должен остановиться и сообщить владельцу, что не может загрузить канонический master-файл.

Запрещено использовать `AGENT.md` как source of truth, если он является redirect-файлом.

## 3. Critical domain

Критичными областями являются:

- расчёт цены;
- расчёт итогов;
- размеры;
- Anwis-формула;
- коммерческое предложение / КП;
- заводские размеры;
- монтаж;
- автообновление;
- release process;
- `releases.json`.

Агент не имеет права менять эти области без:

1. чтения релевантных docs/arc;
2. плана изменений;
3. проверки тестами/сборкой;
4. обновления документации;
5. обновления CHANGELOG.md;
6. явного указания рисков.

## 4. Правило подтверждения владельцем

Файл:

```text
docs/arc/CALCULATION_TEST_CASES.md
```

может содержать статусы кейсов:

- `Подтверждено кодом`
- `Подтверждено владельцем`
- `Требует подтверждения владельца`

Запрещено ставить статус `Подтверждено владельцем`, если владелец явно не подтвердил кейс в чате или задаче.

Если агент видит непроверенный кейс, он должен оставить:

```text
Требует подтверждения владельца
```

---

## 5. Правило четырёх типов размеров

Агент обязан различать 4 типа размеров:

1. **Введённые размеры** — то, что ввёл пользователь.
2. **Расчётные размеры** — размеры после Anwis-формулы.
3. **Заводские размеры** — расчётные размеры минус 20 мм.
4. **Размеры в КП** — хранимые `Width/Height`, то есть расчётные размеры после Anwis-формулы, но до заводского `−20`.

Нельзя смешивать эти размеры.

Особенно важно:

```text
КП берёт Width/Height.
Завод получает ШиринаЗавод/ВысотаЗавод = расчётные −20.
```

## 6. Известная грабля по монтажу

В проекте есть устаревший XML-комментарий в:

```text
OrderItem.Installation.cs
```

Он может утверждать `0 ₽`, но фактическая бизнес-логика кода даёт `500 ₽`.

Правило:

```text
Код важнее комментария.
Комментарий может ввести AI в заблуждение.
Перед правкой монтажа обязательно читать docs/arc/GOTCHAS.md.
```

---

## 7. Release / Auto-update safety

`releases.json` — это рубильник автообновления.

Безопасный порядок:

1. Сначала создать GitHub Release.
2. Сначала загрузить ZIP/артефакт.
3. Проверить, что ZIP доступен для скачивания.
4. Только потом пушить `releases.json` в main.

Запрещено сначала пушить `releases.json`, если GitHub Release/ZIP ещё не готовы.

Иначе пользователи увидят обновление, которое нельзя скачать.

## 8. Универсальный цикл работы агента

Любой агент должен работать так:

```text
Intake -> Context -> Plan -> Execute -> Verify -> Document -> Report
```

### Intake

Понять задачу, критичность и затронутые области.

### Context

Прочитать только релевантную документацию и код.

### Plan

Перед правками написать краткий план.

### Execute

Менять только нужные файлы.

### Verify

Запустить доступные проверки:

```bash
npm run lint
npm run build
npm test
```

или проектные аналоги.

### Document

Обновить:

```text
CHANGELOG.md
docs/arc/CURRENT_STATE.md
релевантные docs/arc/*.md
```

### Report

Финальный ответ должен содержать:

```md
## Сделано

## Изменённые файлы

## Расчётная логика затронута?
- да/нет

## Документация обновлена

## Проверки

## Риски / TODO

## Следующий шаг
```

```text

docs/arc/MULTI_AGENT_ARC_CALC_CONTROL.md

- Не меняй код.
- Не меняй формулы.
- Не ставь «Подтверждено владельцем» без моего явного подтверждения.
- Работай только с документацией.

- docs/arc/MULTI_AGENT_ARC_CALC_CONTROL.md
- docs/arc/CURRENT_STATE.md
- docs/arc/CALCULATION_LOGIC.md
- docs/arc/CALCULATION_TEST_CASES.md
- docs/arc/GOTCHAS.md

Сделай:
1. Выведи краткую таблицу кейсов 1–13.

   - входные данные;
   - ожидаемый результат;
   - текущий статус;
   - что именно должен подтвердить владелец.

   - Case 4: Anwis Проём 1000×1000 -> расчётное 1020×1020 -> заводское 1000×1000.

В финальном ответе дай список вопросов для владельца.
```
