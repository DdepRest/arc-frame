# Auto-Update Periodic Check — Spec

## 1. Motivation

В текущей реализации автообновление проверяется **только при запуске** приложения
(однократный вызов `UpdateService.CheckOnStartupAsync()` из `App.OnStartup`).
Если пользователь работает без перезапуска несколько часов или дней — новые
релизы остаются незамеченными.

Требуется добавить **фоновую периодическую проверку** с детекцией простоя,
чтобы программа сама находила обновления в процессе работы и предлагала их
установить.

---

## 2. Принцип работы (единый комбинированный механизм)

```text
App запущен
    │
    ├──────────────────────────────────────────────────────┐
    │                                                      │
    ▼                                                      ▼
Timer (30 мин)                                    Idle detector (>10 мин)
    │                                                      │
    └──────────────────┬───────────────────────────────────┘
                       │
                       ▼
                 Background check
                 (FetchManifestAsync)
                       │
               ┌───────┴────────┐
               ▼                ▼
          Обновление нет    Обновление есть
               │                │
          Сброс таймера     ┌────┴───────┐
          на 30 мин         │            │
                         Не занят     Занят
                         редактором  (модалка/open)
                            │            │
                            ▼            ▼
                    Показать        Отложить показ
                    диалог          до завершения
                    (модальный)     редактирования
                            │
                     ┌──────┴──────┐
                     ▼             ▼
                Скачать        Отложить
                               (до след.
                                проверки)
```

**Ключевое:** единый механизм. Idle (= 10 мин без ввода) **сбрасывает таймер**
— то есть после idle-паузы проверка происходит сразу, а отсчёт следующих
30 минут начинается заново.

## 3. Триггеры проверки

### 3.1 Таймер (30 минут)

- При старте приложения запускается `DispatcherTimer` с интервалом 30 минут.
- По истечении интервала выполняется тихая фоновая проверка (`isAutomatic = true`).
- После каждой проверки (независимо от результата) таймер сбрасывается на 30 минут.
- Таймер не тикает, если уже идёт проверка или скачивание (`IsChecking` / `IsDownloading`).

### 3.2 Idle-детекция (10 минут)

- Используется WinAPI `GetLastInputInfo` (через P/Invoke) для определения
  времени последнего ввода мыши или клавиатуры.
- Запускается второй `DispatcherTimer` с частотой опроса ~30 секунд (низкая
  нагрузка, достаточная точность).
- Если `GetLastInputInfo` показывает, что ввода не было >10 минут:
  1. Сбросить таймер 30-минутной проверки на 0 (проверка произойдёт сразу
     или в ближайший такт таймера).
  2. Выполнить фоновую проверку.
  3. После проверки перезапустить 30-минутный таймер.
- **Любой ввод** (движение мыши, клик, нажатие клавиши) сбрасывает счётчик
  idle. Программные события (фоновые потоки) не влияют.
- **Исключение:** если приложение свёрнуто (`WindowState == Minimized`) — idle
  не считается, т.к. пользователь всё равно не уведомлён. После
  разворачивания проверка выполняется, если прошло >30 мин с последней.

### 3.3 Ошибка сети / манифеста

- Если `FetchManifestAsync` вернул `null` (нет сети, таймаут, ошибка
  парсинга) при автоматической проверке — **участить** таймер до **5 минут**
  вместо 30.
- После успешной проверки таймер возвращается к 30 минутам.

## 4. Idle Timer Implementation (WinAPI)

```csharp
[System.Runtime.InteropServices.DllImport("user32.dll")]
private static extern bool GetLastInputInfo(ref LASTINPUTINFO plii);

[StructLayout(LayoutKind.Sequential)]
private struct LASTINPUTINFO
{
    public uint cbSize;
    public uint dwTime;
}

private static TimeSpan GetIdleTime()
{
    var plii = new LASTINPUTINFO();
    plii.cbSize = (uint)Marshal.SizeOf(plii);
    if (GetLastInputInfo(ref plii))
    {
        uint idleTicks = (uint)Environment.TickCount - plii.dwTime;
        return TimeSpan.FromMilliseconds(idleTicks);
    }
    return TimeSpan.Zero;
}
```

- Idle-таймер тикает каждые **30 секунд**.
- Если `GetIdleTime() > TimeSpan.FromMinutes(10)` — сброс 30-минутного
  таймера и проверка.

## 5. Диалог обновления (модальный)

### 5.1 Тип и поведение

- Используется **тот же модальный диалог** `DialogService.ShowUpdateAvailable()`,
  что и при ручной проверке.
- Показывается **полный changelog** пропущенных версий (как сейчас).
- Диалог показывается **только если нет активных модальных окон** и
  пользователь не редактирует ячейку DataGrid. Если пользователь занят —
  отложить показ на `DispatcherPriority.Background` до завершения операции.

### 5.2 Кнопки диалога

| Кнопка | Действие |
|--------|----------|
| **Скачать и установить** | Primary, `IsDefault=true`. Начинает скачивание. |
| **Отложить** | GhostButton, `IsCancel=true`. Закрывает диалог. |

### 5.3 Анти-рекомендация (UX)

Рядом с кнопкой «Отложить» добавить поясняющий текст:

```
Отложить  [  Не рекомендуется откладывать обновление надолго  ]
```

Реализация: `TextBlock` с `FontSize=10`, `Foreground={DynamicResource TextMuted}`,
`FontStyle=Italic`. Расположение — под кнопкой «Отложить» или справа от неё
(на усмотрение реализации).

### 5.4 Defer (откладывание)

- При нажатии «Отложить» диалог закрывается.
- `AppSettingsService.SavePendingUpdateVersion(manifest.Latest)` — версия
  обновления сохраняется как «отложенная» (как сейчас).
- **Плашка снова появится** при следующей автоматической проверке (через
  30 мин / idle). Не сохраняется навсегда — только до следующего check-цикла.
- При ручной проверке пользователь увидит отложенную версию так же, как сейчас
  (через `HasPendingUpdate()` и значок на кнопке Settings).

## 6. Взаимодействие с UI

### 6.1 Состояние занятости

Перед показом диалога проверяем:

```csharp
private static bool CanShowDialog()
{
    // 1. Есть ли активные модальные окна?
    if (Application.Current?.Windows.OfType<Window>()
        .Any(w => w.IsActive && w.DialogResult == null) == true)
        return false;

    // 2. Редактируется ли ячейка DataGrid в данный момент?
    //    (Проверка через флаг в MainWindow или через dispatcher)
    //    Можно использовать _isEditingCell флаг, который устанавливается
    //    в BeginningEdit / CellEditEnding событиях.

    return true;
}
```

Если `CanShowDialog() == false` — диалог не показывается. Планируется показ
на `DispatcherPriority.Background` (или через `DispatcherTimer 500ms` retry).

### 6.2 Свёрнутое окно

- Если `WindowState == Minimized` — диалог не показывается.
- При разворачивании (`StateChanged` → `WindowState.Normal`) — проверяем,
  нужно ли показать диалог (если обновление было найдено, пока окно было
  свёрнуто).

### 6.3 Множественные проверки

- Guard: если `IsChecking == true` — новый check-cycle игнорируется.
- Если после проверки прошлый диалог всё ещё открыт (не закрыт) — не
  открывать второй.

## 7. Таймеры (жизненный цикл)

### 7.1 Запуск

`AutoUpdateTimer.Start()` вызывается из `App.OnStartup` после отображения
`MainWindow` (там же, где сейчас `CheckOnStartupAsync`).

Оба таймера (30-минутный и idle-опрос) запускаются сразу.

### 7.2 Остановка

При `Application.Shutdown()` — оба таймера останавливаются.

### 7.3 Сброс таймера

30-минутный таймер сбрасывается в следующих случаях:
- Idle >10 мин обнаружен: таймер сбрасывается на 0.
- После завершения проверки (успех или неуспех): таймер устанавливается на 30 мин.
- После ошибки сети: таймер устанавливается на 5 мин (ускоренный повтор).
- При ручной проверке (пользователь нажал «Проверить обновления»): таймер
  сбрасывается на 30 мин после завершения проверки.

## 10. Сценарии использования

### 10.1 Нормальный сценарий (апдейт найден)

1. Пользователь работает 45 минут.
2. Через 30 мин — фоновая проверка. Найдено обновление.
3. Пользователь не редактирует ячейку → показывается модальный диалог с changelog.
4. Пользователь нажимает «Скачать и установить» → скачивание + установка.
5. Приложение перезапускается.

### 10.2 Обновление отложено

1. Пользователь работает 2 часа.
2. Через 30 мин — проверка. Обновление найдено. Диалог показан.
3. Пользователь нажимает «Отложить».
4. Ещё через 30 мин (60 мин от старта) — снова проверка. Обновление всё ещё
   актуально (версия не изменилась). Диалог показан снова.
5. Пользователь устанавливает.

### 10.3 Idle-триггер

1. Пользователь ушёл на перерыв (15 минут).
2. Idle-детектор (опрос каждые 30 сек) замечает, что прошло >10 мин без ввода.
3. 30-минутный таймер сбрасывается на 0 → фоновая проверка.
4. Диалог не показывается, т.к. пользователя нет. (Или показывается —
   пользователь увидит его, когда вернётся и сделает первый ввод.)
5. **Решение:** при idle-триггере проверка выполняется, но диалог
   показывается только после того, как пользователь сделает первый ввод
   (сброс idle). Или сразу — и пользователь увидит его при возвращении.
6. **Выбор реализации:** диалог показывается сразу после проверки (даже при
   idle). Пользователь увидит его при возвращении. Если окно свёрнуто —
   диалог покажется при разворачивании.

### 10.4 Ошибка сети

1. Через 30 мин — проверка. Сеть недоступна.
2. Таймер переключается на 5-минутный интервал.
3. Через 5 мин — повтор. Сеть снова недоступна.
4. После 3-4 неудачных попыток — таймер возвращается к 30 мин (чтобы не
   спамить запросами, если сеть стабильно недоступна).

### 10.5 Пользователь редактирует ячейку

1. Проверка прошла успешно, обновление найдено.
2. `CanShowDialog()` возвращает `false` (пользователь редактирует ячейку).
3. Через 500 мс — повторная проверка `CanShowDialog()`.
4. Как только редактирование завершено — диалог показывается.

## 11. Отличия от ручной проверки (isAutomatic)

| Аспект | Авто (таймер/idle) | Ручная |
|--------|-------------------|--------|
| Toast "Проверка..." | ❌ | ✅ |
| Toast "Обновлений нет" | ❌ | ✅ |
| Toast ошибка манифеста | ❌ (молча) | ✅ Warning |
| Toast ошибка скачивания | ✅ Error | ✅ Error (MessageBox) |
| Диалог с changelog | ✅ (с анти-реком.) | ✅ (без анти-реком.) |
| Отложить | ✅ до след. проверки | ✅ до след. запуска |
| Учащение при ошибке | да (+ 5 мин) | нет |

Анти-рекомендация добавляется **только в автоматическом режиме**. При ручной
проверке текст «Не рекомендуется откладывать» не показывается.

## 13. Класс UpdateCheckTimer

Предлагается вынести логику таймеров в отдельный класс
`UpdateCheckScheduler` (чтобы не раздувать `UpdateService`).

```csharp
public class UpdateCheckScheduler
{
    private DispatcherTimer? _checkTimer;          // 30 мин / 5 мин
    private DispatcherTimer? _idlePollTimer;       // 30 сек
    private DateTime _lastCheckTime;
    private int _failedConsecutiveChecks;

    public event EventHandler? CheckTriggered;

    public void Start();
    public void Stop();
    public void ResetTimer(TimeSpan interval);
    
    private void OnCheckTimerTick();
    private void OnIdlePollTimerTick();
}
```

Либо оставить в `UpdateService` как `static` поля — на усмотрение
реализации. Важно: таймеры запускаются и живут в UI-потоке
(`DispatcherTimer`), т.к. по истечении требуется обновить UI.

## 14. Риски

| Риск | Описание | Митигация |
|------|----------|-----------|
| **Высокая частота запросов** | При частых ошибках сети — 5-минутный интервал может генерировать много запросов | Ограничение: после 4 неудач → 30 мин |
| **Idle-ложные срабатывания** | Если пользователь читает PDF/смотрит видео вторым монитором, программа считает idle | Используется системный `GetLastInputInfo` — он честно отражает отсутствие ввода |
| **Конфликт с ручной проверкой** | Ручная проверка + таймер срабатывают одновременно | Guard `IsChecking` блокирует повтор |
| **Производительность** | Каждые 30 сек опрос WinAPI | `GetLastInputInfo` — очень лёгкий вызов (наносекунды) |
| **Несовместимость WinAPI** | `GetLastInputInfo` доступен на всех NT-системах | Минимальная версия: Windows XP. Проект уже использует WinAPI |
