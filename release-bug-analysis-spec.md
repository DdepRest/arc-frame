# Spec — Анализ на баги перед релизом

> **Статус:** Интервью завершено, spec утверждён  
> **Дата спека:** 2026-06-28  
> **Затронутые версии:** v3.37.2 (ретроспектива) + следующий Unreleased  
> **Формат выходного артефакта:** Структурированный список багов с указанием места (файл/класс/строка), уровнем серьёзности и краткой рекомендацией

---

## 1. Цель

Провести комплексный анализ кодовой базы и проектной документации MosquitoNetCalculator на предмет багов, регрессий, визуальных несоответствий и документарных расхождений **перед следующим релизом**, а также ретроспективно проверить текущую версию v3.37.2.

---

## 2. Скоуп анализа

### 2.1. Версии

| Версия | Что проверяем |
|--------|--------------|
| **v3.37.2** (текущая) | Ретроспектива: не пропущены ли баги в уже выпущенном коде; корректность последних фиксов (SelectAll race, LostFocus clamp, DeleteRowButton padding, UpdateLog sort-in-code, ParseVersion диагностика) |
| **Unreleased** (следующий) | Всё, что попадёт в следующий релиз: проверка готовности кода, отсутствие WIP-элементов, консистентность версий и документации |

### 2.2. Подходы (все применяются)

1. **Code review** — систематический обход известных опасных мест (`docs/arc/GOTCHAS.md`) + аудит нового/изменённого кода с момента v3.37.0.
2. **Ручное функциональное тестирование** — пошаговая проверка UI и расчётов по сценариям использования.
3. **Статический анализ** — проверка на dead code, неиспользуемые using'и, подозрительные паттерны, утечки событий, race conditions.
4. **Документарный аудит A.R.C.** — актуальность `CURRENT_STATE.md`, синхронизация `CHANGELOG.md` ↔ `update-log.json`, версия в `.csproj`, консистентность `releases.json`.

## 3. Приоритетные области

### 3.1. UI/UX (высший приоритет по запросу владельца)

> **Известная проблема от владельца:** «Не все места скруглённые, особенно в Расчёте». Конкретно — таблица (DataGrid) с добавленными товарами.

**Что проверить:**
- Скругление углов (`CornerRadius`) в `OrderItemsControl.xaml` — строки таблицы, ячейки, заголовки, вспомогательные элементы.
- Стилистическое единообразие между `OrderItemsControl`, `PricesControl`, `QuickAddControl`, `TotalCardControl`.
- Корректность отображения `AmountOnlyProducts` (Брус, Пояс, Доставка) — скрыты ли колонки Кол-во/Площ./Дл., отключены ли поля в QuickAdd.
- Анимации фокуса, hover-эффекты, выделение текста (`SelectAll` синхронный, без `BeginInvoke`).
- `LostFocus` на Ширине/Высоте — формула Anwis не перехватывает набор.
- `PropertyChanged` на Цене — автоширина колонки корректно трекает набор.
- TitleBar-полоска прогресса скачивания (`UpdatesTabControl`, `MainWindow`) — видимость, цвет, анимация fade-in/fade-out.
- Диалог «Доступно обновление» — changelog, ScrollViewer, кнопки, прогресс внутри диалога.

### 3.2. Расчёты и формулы (критично)

- Утечка формул Anwis на не-Anwis товары (`GOTCHAS#1`) — регрессионная проверка после v3.35.0.
- `SetAnwisModeQuiet` при загрузке/клонировании (`GOTCHAS#2`) — `OrderItem.Dto`, `CalculationViewModel.LoadFromOrderData`, копирование заказов.
- Монтаж с `Quantity > 1` (`GOTCHAS#12`) — `TotalWithDeduction` умножает deduction на Quantity для режимов 1 и 2.
- Эталонные расчётные кейсы (`CALCULATION_TEST_CASES.md`, Case 1–16) — все значения должны совпадать с кодом.
- `AmountOnlyProducts` — корректность расчёта (только сумма, без Ш/В/Кол-во).

### 3.3. Сохранение/загрузка данных (критично)

- `CalculatedValue` и `Total` НЕ сохраняются в JSON (`GOTCHAS#3`).
- Пути к данным — только `%AppData%`, не `BaseDirectory` (`GOTCHAS#9`).
- Undo/Redo — `UnsubscribeAll` перед `RestoreFromSnapshot`, нет утечек событий (`GOTCHAS#10`).
- Копирование заказов — `GenerateCopyContractNumber`, новый номер, статус «Новый», дата.
- Экспорт/импорт JSON — дубликаты не создаются, формат корректен.

### 3.4. Печать КП (высокий)

- HTML-экранирование всех строковых полей (`GOTCHAS#6`) — имя, адрес, примечания, названия товаров.
- Размеры в КП — расчётные (`Width`/`Height`), а не введённые (`ШиринаВвод`/`ВысотаВвод`) — `CALCULATION_TEST_CASES#Case 11`.
- Сумма прописью — `AmountInWordsService`.
- Доп. КП в печати — блок с номерами и суммами.
- Монтаж «В констр.» — корректное отображение в таблице печати.

### 3.5. Автообновление (высокий)

- `releases.json` синхронизирован с `.csproj` (`GOTCHAS#5`).
- `update-log.json` содержит записи для всех версий.
- `GetAvailableUpdate` корректно сравнивает версии, `ParseSafe` обрабатывает edge cases.
- TitleBar-полоска прогресса — корректная работа `IsDownloading`, `DownloadProgress`.
- Toast-фильтрация — при `isAutomatic=true` нет спама, при ручной проверке — информативные сообщения.
- SHA-256 проверка, watchdog .bat, перезапуск.

### 3.6. Темы и стили (средний)

- Frozen brush при переключении темы — `ThemeService` (`GOTCHAS#7`).
- Все цвета из `DynamicResource` — нет хардкоженых цветов, которые ломаются при смене темы.
- Корректность отображения в светлой и тёмной темах.

### 3.7. Цены и миграции (критично)

- `ApplyMigrations` — новые товары добавлены и в `DefaultPrices`, и в миграцию (`GOTCHAS#4`).
- `RemoveAll` vs `prices = prices.Where(...).ToList()` — изменения не теряются.
- `prices.json` — корректный формат, читаемость, сохранение пользовательских цен.

## 4. Документарный аудит A.R.C.

> **Решено в интервью:** Документация = часть качества релиза. Проверить все `docs/arc` файлы.

**Что проверить:**
- `CURRENT_STATE.md` — актуальная версия (должна быть 3.37.2), статус последних изменений, открытые вопросы.
- `CHANGELOG.md` — `Unreleased` не пустой (если готовится релиз), все изменения v3.37.2 описаны.
- `MosquitoNetCalculator/Resources/update-log.json` — синхронизация с `CHANGELOG.md` (количество записей, версии, даты).
- `releases.json` — `latest` совпадает с `.csproj`, записи для v3.37.2 присутствуют, SHA-256 и URL корректны.
- `MosquitoNetCalculator/MosquitoNetCalculator.csproj` — поле `<Version>` является единственным источником правды.
- `GOTCHAS.md` — все 15 грабель актуальны, нет устаревших предупреждений, новые фиксы задокументированы.
- `CALCULATION_TEST_CASES.md` — все 16 кейсов подтверждены, статусы корректны.
- `TESTING_CHECKLIST.md` — **обновить** с v3.24.0 до актуального состояния (v3.37.2+): добавить разделы про AmountOnlyProducts, копирование заказов, обновлённую систему автообновления, TitleBar-прогресс, SelectAll/LostFocus поведение.
- `validate-docs.ps1` — запустить, проверить 8 проверок на ошибки.
- `SYMBOL_INDEX.md` / `INTENTS.md` — актуальность после последних изменений.

## 5. Визуальный аудит (конкретика от владельца)

> **Конкретная жалоба:** «Не все места скруглённые. Особенно в Расчёте. Таблица, где показаны добавленные товары».

**Что проверить в `OrderItemsControl.xaml` / `MainWindow.xaml`:**
- `DataGrid` — `CornerRadius` у строк (RowStyle), ячеек (CellStyle), заголовков (ColumnHeaderStyle).
- `DataGridRow` — проверить, что `Border` внутри `ControlTemplate` имеет `CornerRadius`, а не прямоугольные углы.
- `QuickAddControl` — карточка быстрого ввода: `CornerRadius` текстовых полей, комбобоксов, кнопки «Добавить».
- `TotalCardControl` — итоговая карточка: `CornerRadius` внешнего `Border`.
- `ActionBarControl` — кнопки в верхней панели: единообразие `CornerRadius`.
- `SidebarControl` — боковая панель заказчика: скругления полей ввода.
- Несоответствие между `CardStyles.xaml` и фактическими контролами — если `CardStyle` предполагает `CornerRadius="12"`, но конкретный контрол его не применяет.

**Референс:** `UpdatesTabControl.xaml` — там карточки обновлений имеют `CornerRadius="12"` и считаются эталоном. Все остальные контролы должны быть к ним визуально согласованы.

## 6. Критерии серьёзности

> **Решено в интервью:** Все уровни — от критических до низких (визуальные мелочи, опечатки).

| Уровень | Описание | Примеры |
|---------|----------|---------|
| **КРИТИЧНЫЙ** | Блокер релиза. Неправильные расчёты, потеря данных, краш. | Утечка Anwis, сохранение derived-полей, неэкранированный HTML, неправильные заводские размеры |
| **ВЫСОКИЙ** | Серьёзный UX-дефект или функциональная ошибка. | Toast-спам, некорректный прогресс обновления, потеря пользовательских цен |
| **СРЕДНИЙ** | Некритичный баг, workaround есть. | Неправильный `UpdateSourceTrigger`, неактуальный чек-лист, рассинхронизация документации |
| **НИЗКИЙ** | Визуальная мелочь, опечатка, несоответствие стиля. | Отсутствие `CornerRadius`, несогласованные отступы, устаревший комментарий |

## 7. Порядок выполнения анализа

1. **Документарный аудит** (быстрый win): запустить `validate-docs.ps1`, проверить версии, CHANGELOG, update-log, releases.json.
2. **Визуальный аудит UI** — открыть приложение, проверить скругления в «Расчёте», единообразие стилей, анимации.
3. **Расчётные регрессии** — прогнать все 16 кейсов из `CALCULATION_TEST_CASES.md` через приложение.
4. **Code review по GOTCHAS** — проверить все 15 опасных мест на отсутствие регрессий.
5. **Ручное тестирование по чек-листу** — пройти обновлённый `TESTING_CHECKLIST.md`.
6. **Статический анализ кода** — dead code, неиспользуемые using'и, подозрительные паттерны.
7. **Составление отчёта** — структурированный список с уровнями серьёзности и рекомендациями.

## 8. Edge cases для проверки

| Сценарий | Где проверять |
|----------|--------------|
| Пустой заказ (0 позиций) | `CalculationViewModel.CalculateTotal`, печать КП, «На завод» |
| Quantity = 0 или отрицательное | `OrderItem` setter'ы, `CalculationViewModel.AddItem` (qty <= 0 → 1) |
| Очень большое Quantity (1000+) | `TotalWithDeduction` clamp, переполнение, производительность DataGrid |
| Ширина/Высота = 0 для Anwis | `AnwisSize.ОтВвода` — `max(0, raw−30)` → reverse = 30? |
| Клонирование заказа с Anwis ББ 70 | `SetAnwisModeQuiet`, корректность `ШиринаВвод`/`ВысотаВвод` |
| Импорт старого JSON (до v3.22) | `ApplyMigrations`, отсутствие `CalculatedValue`/`Total` в DTO |
| Автообновление без интернета | `FetchManifestAsync` → null → молчаливый выход |
| Диалог обновления + закрытие окна приложения | `Owner` relationship, корректное закрытие диалога |
| Переключение темы во время редактирования ячейки | Frozen brush, `ThemeService`, потеря фокуса |
| Быстрое добавление 50+ позиций | Производительность DataGrid, Undo/Redo stack |
| Печать с символами `< > & "` в адресе | `EscapeHtml`, `PrintService.FillTemplate` |

## 9. Ожидаемый выходной артефакт

Структурированный markdown-файл (`bug-analysis-report.md` или аналогичный) со следующими колонками:

```
| # | Уровень | Модуль | Файл | Описание | Рекомендация |
```

- **#** — порядковый номер
- **Уровень** — КРИТИЧНЫЙ / ВЫСОКИЙ / СРЕДНИЙ / НИЗКИЙ
- **Модуль** — Расчёты / UI / Автообновление / Печать / Сохранение / Документация
- **Файл** — конкретный файл и строка (если применимо)
- **Описание** — что именно не так
- **Рекомендация** — краткое руководство по исправлению или ссылка на GOTCHAS/спек
